# Hermes 工具选型

> 本文档记录 Hermes 项目在不同任务场景下的技术栈决策。
> **任何新 PR / 新模块在动手前应先核对此文档。**
> 如果出现新决策推翻旧决策，请直接更新本文件并更新 `last_updated`。

---

## 1. 分层定位

| 层 | 职责 | 主力技术栈 |
|---|---|---|
| 数据采集层（ingestor） | 各平台监听 / 轮询 / 抓取，输出统一 captures markdown | **Python**（长连接必需场景） / **n8n**（轻量轮询场景） |
| 核心处理层 | LLM 调用、token 预算、批处理、ML、特征工程 | **Python** |
| 工作流粘合层 | "X 发生 → 做 Y → 通知 Z" 类流程 | **n8n** |
| Agent 原型层 | 单 agent / 简单 multi-step，重在快速验证 | **n8n AI Agent 节点** |
| Agent 生产层 | 复杂多 agent / 高频策略执行 / 严格 SLA | **Python**（LangGraph 或自写 runtime） |
| 数据存储层 | captures markdown / 元数据 / 向量索引 | 文件 + sqlite，必要时再加向量库 |

---

## 2. 决策矩阵

### 2.1 何时选 Python

- Telegram / 类似平台的 **MTProto 长连接**（n8n 不支持）
- LLM 调用涉及复杂 **token 预算控制 / retry / prompt 版本管理**
- **批处理大量 markdown / JSON 文件** 的脚本
- ML 任务：MinHash、向量化、特征工程
- 实盘 / 高频任务的 **订单管理 / 风控**
- 任何需要严格 **单元测试覆盖** 的核心算法

### 2.2 何时选 n8n

- 文件落地后的 **分发**：推 Telegram / Discord / 邮件 / 微信 / Slack
- B2B **业务流程**：表单 → CRM → 邮件跟进 → 提醒
- **个人自动化**：邮件分流、日程、通知路由
- 第三方 SaaS **粘合**：Notion、Sheets、Airtable、HubSpot
- **可视化 debug 重要的 agent 原型**
- 任何 "X → Y → Z" 形态、用 Python 反而显冗长的流程
- **Reddit / TikTok Ads Library / 一般 HTTP API** 这类轻量轮询型 ingestor

### 2.3 何时**不要**用 n8n

- Telegram 群消息监听（n8n 只支持 Bot API，读不了非管理员群）
- 高频 / 高 QPS / 大批量并发处理
- 需要严格单元测试的核心算法
- 实盘交易订单提交
- 复杂 prompt 版本管理 + 严格 token 预算

### 2.4 Agent 选型阶梯

1. **第一个 agent**：n8n AI Agent 节点。可视化执行历史是 agent debug 的救命特性。
2. **第二、第三个 agent**：仍优先 n8n。除非显示出明确瓶颈。
3. **顶不住才升级到 Python**：上 LangGraph、AutoGen 或自写 runtime。
4. **永远不要**：第一个 agent 就直接上 Python multi-agent framework。复杂度爆炸、debug 痛苦。

## 4. n8n 部署规范

- **容器名**：`hermes-n8n`
- **VPS 路径**：`/opt/hermes-blueprint/n8n-data/`
- **网络**：内网或 nginx 反代 + basic auth，**不要裸暴露公网**
- **环境变量隔离**：
  - n8n 自己的 `N8N_ENCRYPTION_KEY` 必须独立保管
  - **不要**与 `hermes-bridge-ingress` 的 Telegram secrets 混在同一个 `.env`
- **备份**：n8n workflow 定期 export 到仓库 `90-Ops/n8n-workflows/*.json`（不含敏感字段）
- **依赖**：n8n 升级走 docker-compose pin 版本，**不要 latest tag**

## 5. 写新模块前的"三问"自省清单

| 问题 | 答案 → 选择 |
|---|---|
| 这件事的核心是 "处理算法" 还是 "流程粘合"？ | 算法 → Python；粘合 → n8n |
| 这件事是 "一次性写完跑很久" 还是 "未来经常增删步骤"？ | 写完跑很久 → Python；经常改 → n8n |
| 这件事的 debug 痛点是 "性能" 还是 "看不见状态"？ | 性能痛 → Python；看不见痛 → n8n（可视化） |

三个问题里 **任意两个指向 n8n**，就用 n8n。

## 7. 反模式（不要做）

- ❌ 用 Python 写 Phase 2 的"客户邮件跟进流程"——应该用 n8n
- ❌ 用 n8n 写 Phase 1 的 digest 核心 LLM 循环——应该用 Python
- ❌ 用 n8n 监听 Telegram 群（只能读 Bot 加入的频道）
- ❌ 第一个 agent 直接上 LangGraph multi-agent
- ❌ n8n 跟 Telegram bridge 共用同一个 `.env`
- ❌ 在 Python 核心里硬编码 prompt，不留 prompt 版本管理

## 8. LLM 选型

### 8.1 主力：DeepSeek

- **模型**：`deepseek-chat`（日常）/ `deepseek-reasoner`（需推理时）
- **入口**：官方 API
- **成本量级**：deepseek-chat ≈ \$0.14 / 1M input、\$0.28 / 1M output（参考 2026 年价位，以官方为准）
- **优势**：中文好、价格低、Phase 1 月预算 cap \$5 够用
- **劣势**：reasoning 弱于 GPT-4o / Claude；agent 多轮工具调用稳定性弱于 Claude Sonnet

### 8.2 何时切到别的 LLM

| 场景 | 备选 | 切换理由 |
|---|---|---|
| Agent 复杂多步骤工具调用 | Claude Sonnet | 工具调用稳定性最佳 |
| 长上下文（>64k tokens） | Gemini 2.x | 上下文窗口 |
| 严格 JSON / structured output | GPT-4o-mini | structured output 成熟度 |
| 极敏感数据需本地处理 | 本地 Qwen / DeepSeek-V2 | 不出 VPS |

切换必须通过 `llm_client.py` 的 provider 抽象，**禁止业务代码直接 hardcode 模型名**。

### 8.3 多 LLM 路由（Phase 5 决策）

启动 60-Toolbox v0 时再决：自建 router / OpenRouter / 各自直连。

## 10. 测试与版本管理

### 10.1 Python（核心层）

- **单元测试**：pytest，外部 IO（LLM / 网络 / 文件 / Telegram）一律 mock
- **CI**：GitHub Actions（已有 `audit-script-hygiene.yml`，后续补 pytest workflow）
- **分支**：`feat/*`、`fix/*` + PR review，绝不直接 push main
- **测试时长硬约束**：单文件 ≤ 5 秒，整个 suite ≤ 30 秒（来自 PR #29 卡死教训）
- **异步测试**：必须 `asyncio.timeout` 包裹，禁止真实 `run_until_disconnected`、长 sleep

### 10.2 n8n（粘合层 + agent 原型）

- **workflow 版本管理**：导出 JSON 到 `90-Ops/n8n-workflows/*.json`，进 git
- **导出前必须替换敏感字段**为占位符（API key / webhook secret / chat_id）
- **每个 workflow** 必须保留 disabled 的"测试触发器" + 测试输入示例
- **月度 review**：检查 executions 失败率，>5% 必须修；>10% 评估升级到 Python

### 10.3 Prompt 版本管理

- Prompt 不写在代码里，落到 `50-Intelligence/prompts/{task}/v{N}.md`
- 切换通过环境变量或配置文件，不改代码
- 每次升级 prompt 必须保留旧版本，便于 A/B 对比

## 11. 升级路径（n8n agent → Python agent）

n8n AI Agent 节点是起点，**不是终点**。触发升级的条件：

| 触发条件 | 行动 |
|---|---|
| 单次 agent 调用 > 90 秒 | 评估升级 |
| 单 agent 工具数 > 8 | 升级 |
| 需要并发 > 5 个 agent 实例 | 升级 |
| 需要 streaming 输出给用户 | 升级 |
| 需要严格 JSON schema 验证 | 升级 |
| n8n executions 失败率 > 10% 且无法 debug | 升级 |
| 需要复杂的 retry / circuit breaker | 升级 |

**升级阶梯**：n8n AI Agent → LangGraph (Python) → 自写 runtime。
**禁止跳级**：不要一上来就自写 runtime，那是 over-engineering。

## 12. 已知坑（教训累积）

| 坑 | 出现于 | 教训 |
|---|---|---|
| 异步测试卡死 (`run_until_disconnected`) | PR #29 | 所有异步测试必须 `asyncio.timeout`；mock 长连接行为 |
| dataclass 动态导入 NoneType | PR #27 | `importlib` 后必须 `sys.modules[name] = mod`；或避免连字符文件名 |
| 大写符号被小写化（`SOL` → `sol`） | PR #27 | token 候选项保留原始 case，专用正则提取大写符号 |
| `/healthz` 返回 ok 但监听器掉线 | PR #29 前 | 健康检查必须反映"业务健康"，不能只反映"进程活着"；用 503 + degraded |
| 3 字母大写符号被 stopword 过滤 | PR #27 | 符号白名单优先于通用过滤 |
| ToS 灰色爬虫账号被 ban（预防） | (未发生) | burner 账号 + 备份方案，永不依赖单点 |

**新坑加入流程**：发现新坑 → 修复 PR → 在本表登记 → 必要时更新 `60-AI-Toolbox/README.md` 的启动 checklist。
