# Hermes 平台扩展模式

> 本文档定义"如何往 Hermes 加一个新数据源"的统一规范。
> Phase 1 已就位 Telegram bridge；任何后续平台（Reddit / LinkedIn / TikTok / Discord / Feishu / 自建 webhook / RSS / ...）都按本文档接入。

---

## 1. 设计哲学

**Bridge Pattern** —— 任意来源 → 各自的 ingestor → 统一格式 markdown captures → 统一下游管道。

```
┌──────────────────────────────────────────────────────────┐
│                       上游（每平台一份 ingestor）          │
├──────────────────────────────────────────────────────────┤
│ Telegram bridge   Reddit bridge   LinkedIn bridge   ...  │
│ (Python+Telethon) (n8n)           (n8n+Phantombuster)    │
└──────────┬───────────────┬──────────────────┬────────────┘
           │               │                  │
           ▼               ▼                  ▼
   ┌─────────────────────────────────────────────────────┐
   │  /data/inbox/{Source}-Captures/{date}/{id}.md       │
   │  统一格式（见 §3 契约）                              │
   └──────────────┬──────────────────────────────────────┘
                  │
                  ▼
   ┌─────────────────────────────────────────────────────┐
   │            下游（一份覆盖所有源）                    │
   │  rsync → inbox-stats → extract-keywords →            │
   │  digest → 雷达 → 共振 → 研究 agent                   │
   └─────────────────────────────────────────────────────┘
```

**核心承诺**：只要 ingestor 守住 §3 契约，**下游一行代码不用改**就支持新平台。

## 3. ingestor 统一契约（必须遵守）

### 3.1 文件路径

```
/data/inbox/{Source}-Captures/{YYYY-MM-DD}/{id}.md
```

- `Source` 用首字母大写：`Telegram` / `Reddit` / `LinkedIn` / `TikTok` / `Discord` / `Feishu` / `Web` / `Repo` / `RSS` / `Twitter`
- `id` 由 ingestor 生成，要求**幂等**（同一条原始消息重复处理 → 同一文件名）：
  - Telegram: `{chat_id}-{message_id}`
  - Reddit: `{subreddit}-{post_id}` 或 `{subreddit}-{post_id}-{comment_id}`
  - Discord: `{guild_id}-{channel_id}-{message_id}`
  - LinkedIn: `{post_urn}` 或 hash(url)
  - TikTok: `{video_id}` 或 `ads-{ad_id}`
  - Twitter: `{tweet_id}`
  - RSS: hash(url) + 短 hash(title)

### 3.2 frontmatter 必备字段

```yaml
---
source: telegram | reddit | linkedin | tiktok | discord | feishu | web | repo | rss | twitter
source_id: <ingestor 生成的稳定 id>
timestamp_utc: 2026-05-21T08:30:00Z
url: <原始链接，无则 null>
author: <发布者用户名 / handle>
chat_or_channel_id: <群 / 频道 / subreddit / board / list 的 id>
chat_or_channel_name: <人类可读名称>
language: zh | en | ... | unknown
---
```

可选字段（按需）：

- `reply_to`：父消息的 source_id
- `media`：list of media URLs
- `tags`：list of strings
- `geo`：`{country, city}`
- `engagement`：`{views, likes, reposts, comments}`（适用于 Reddit / TikTok / LinkedIn / Twitter）
- `is_repost`：bool
- `original_source_id`：转发原文的 source_id

### 3.3 正文格式

- 第一段：原始内容（保留原文，不做翻译 / 摘要）
- 可选第二段：用 `---` 分隔后放 ingestor 的预处理（翻译 / 摘要 / 标签）
- **不允许**在正文混入未在 frontmatter 声明的 metadata；所有 metadata 一律走 frontmatter

### 3.4 编码与命名

- 文件编码：**UTF-8 无 BOM**
- 换行：**LF**
- 文件名禁止空格、中文、特殊字符（`/ \ : * ? " < > |` 等）；推荐 `[a-zA-Z0-9_-]+`
- markdown 内允许任何 UTF-8

### 3.5 写入语义

- ingestor 必须**先写临时文件再原子 rename**，避免下游读到半截文件
- 重复写入同一 source_id 应是幂等（覆盖 OK，但内容应一致）
- ingestor 自身**不删旧文件**；保留 / 清理是单独的 retention 任务

### 3.6 不允许在 ingestor 里做的事

- ❌ 调用 LLM 做处理（处理是下游 digest 的职责）
- ❌ 做"内容过滤"（能进 captures 就全部进，过滤交给下游 pipelines）
- ❌ 做"翻译 / 摘要"以外的 enrichment（保持 ingestor 简单）
- ❌ 写到非自己平台的 captures 目录

## 5. 推荐落地顺序

### Group A：高 ROI 低成本（Phase 1 收尾后 / Phase 2 启动后立即做）

1. **Reddit**（n8n 半天）—— B2B + Crypto 双收益
2. **RSS 新闻源**（n8n 半天）—— Phase 4 事件驱动信号
3. **TikTok Ads Library**（n8n + HTTP 1 天）—— Phase 2 竞品广告研究
4. **浏览器剪藏**（自建扩展 2 天）—— 个人收藏整合到 inbox

### Group B：明确需求触发后做

5. **Discord**（n8n 1 天）—— 部分 crypto 社区只在 Discord
6. **Twitter / X**（官方 API 或 Apify 2 天 + 持续费用）—— Phase 3 / Phase 4 必备
7. **Repo Captures（GitHub webhook）**（自建 1 天）—— Phase 5 之后做"仓库研究 agent"时用

### Group C：Phase 2 推广通道决策后再做

8. **LinkedIn**（Sales Nav + Phantombuster 3 天 + 持续费用）—— Phase 2 高价值

### Group D：可能永远不做

9. **TikTok 内容监听** —— 除非 B2B 内容营销策略明确需要，否则路径 A (Ads Library) 已够

## 6. 新平台接入 checklist

每加一个新平台，按顺序勾：

- [ ] 1. 在本文档 §4 表里登记难度评分、月成本
- [ ] 2. 决定 ingestor 技术栈（Python / n8n / 第三方 SaaS 包装）
- [ ] 3. 在 `00-Inbox/{Source}-Captures/.gitkeep` 占位（已有不用建）
- [ ] 4. 实现 ingestor，**严格遵守 §3 契约**
- [ ] 5. 写自检 / 单元测试：
  - [ ] frontmatter 必备字段齐全
  - [ ] `id` 幂等（同一原始消息 → 同一文件）
  - [ ] 时间戳为 UTC
  - [ ] 文件名合规
  - [ ] 写入是原子操作（temp → rename）
- [ ] 6. 在 `00-Inbox/Inbox-Processing-Rules.md` 登记新源
- [ ] 7. VPS 部署 + 跑 24h 观察
- [ ] 8. 验证下游 inbox-stats / extract-keywords / digest 能识别新源
- [ ] 9. 不要触碰 secrets：API key / token 一律走环境变量；测试用 mock
- [ ] 10. PR 描述里写清楚月成本估算 + 合规风险等级

## 7. 合规与风险

### 7.1 严守底线

- ❌ **不爬取需要登录态的私密内容**（除非账号 owner 明确授权）
- ❌ **不绕过 robots.txt**
- ❌ **不规避反爬虫**（意味着同意 ToS 不允许的行为）
- ❌ **不大规模复制版权内容**（fair use 范围内的元数据 / 摘要 OK）
- ❌ **不存储他人 PII** 超出必要（用户名 / handle 可以，地址 / 电话 / 邮箱不要）

### 7.2 风险分级处理

| 风险等级 | 平台示例 | 处理 |
|---|---|---|
| 低 | Reddit / RSS / 公开 API | 直接做 |
| 中 | TikTok 内容 / Twitter Apify | burner 账号 + IP 池 + 频率克制 |
| **高** | LinkedIn 自动化 | burner 账号 + 接受被 ban 的可能 + 有备份方案 |

### 7.3 数据保留

- captures 默认本地保存（VPS + Windows）
- **不上传到公开 git 仓库**（已 `.gitignore`）
- 涉及 PII 的字段在进数据库 / 向量库前做脱敏
- 默认保留 90 天，超期归档到外部冷存储（Phase 5 之后再决策）

## 8. 反模式（不要做）

- ❌ 同一平台多个 ingestor 并行写同一个 `{Source}-Captures/` 目录（会互相覆盖）
- ❌ ingestor 直接调 LLM 做处理（处理是下游 digest 的职责）
- ❌ frontmatter 字段名使用驼峰 / 中文（必须 snake_case 英文）
- ❌ `id` 用时间戳生成（不幂等，重启会重复）
- ❌ 在 ingestor 里做"过滤"（能进 captures 就全部进）
- ❌ 一个 ingestor 同时支持多个平台（每个平台一份独立 ingestor）
- ❌ ingestor 直接读写真实 secret 文件，不走环境变量
